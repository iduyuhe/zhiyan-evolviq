二、核心交付与实证

V1-1 · 跨 Agent 知识图谱

  • 把 11 个 Agent 的工业语义（物料 / 设备 / 产品 / 缺陷 / 产线 / 部件 / 订单 / 元件 / 标准 / 变更 / 良率）桥接成一张跨域语义网；质量案例经设备类型关键词自动桥接 PM 设备与部件；引擎执行结果以 fire-and-forget 方式增量写入图谱。
  • 种子构建 102 节点 / 95 边；Neo4j 不可达时自动回退内存图（dict 邻接表），生产可达自动切换 Neo4j 模式，零启动中断。
  • 前端：知识图谱 Tab（统计 + 按类型/邻居查询 + 跨 Agent 桥接示例 + 力导向图可视化）。

V1-2 · 全 11 Agent MCP 能力层联邦

  • 采用「能力层联邦」——统一 MCP server 对外暴露 11 Agent 的 in-process 工具，Agent 内部仍走 in-process 主路径，与 T4「收敛删死代码」方向一致，零运行时风险。
  • {agent}__{method} 命名空间统一注册 38 个工具（供应链 6 / 设备 4 / 良率 4 / 其余 8 Agent 各 3）；HTTP + stdio 双传输均从联邦注册表驱动；保留 6 个旧版供应链工具短名别名以兼容既有 MCP 客户端。
  • 11/11 Agent 代表工具经真实 API 全 200；未知工具正确 404；集成测试 /mcp/tools 6 工具契约保持不变。

V1-3 · 工业协议网关齐备

  • 四类工业协议网关（Modbus / MQTT / OPC-UA / IPC-CFX）均实现 BaseGateway 统一接口（read / write / connect / health）；GatewayManager 统一持有并 best-effort 初始化；/gateways API 暴露健康总览、单网关详情与实时读数。
  • 沙箱无真实 OPC-UA Server / AMQP Broker 时，四类网关自动 simulated 模式，读数正常、绝不破管；附带收益——Modbus / MQTT 经 GatewayManager 首次获得运行时可观测能力，消除原死代码隐患。
  • 前端：网关 Tab（4 类网关健康卡 + 实时读数轮询面板）。

V1-4 · 控制台「按效果调参」

  • StrategyTuner 汇聚「放权度（自主率）vs 人类信任度（介入批准率/驳回）」，生成调参建议（放权 / 收紧）；采纳建议或手动调参后实时改写运行时授权边界并留审计轨迹。
  • 关键修复：补齐漏掉的 yield_analysis 第 11 个授权边界，使 11/11 Agent 均有授权边界。
  • supply_chain 置信阈值 0.8 → 0.72 运行时真正改写；调参夹紧区间 [0.5, 0.95]；非法参数 400 / 未知 Agent 404；审计轨迹可查。
  • 前端：策略调参 Tab（旋钮 + 效果一页视图、建议采纳、手动调参、审计轨迹）。
铁律：调参引擎只调整策略阈值，绝不改写任何业务数字或动作。

四、架构与韧性说明（升级要点）

  • 韧性降级铁律：所有外部依赖（PostgreSQL / Neo4j / OPC-UA Server / AMQP Broker / MCP 客户端库）均惰性导入 + 不可达自动回退本地替代（SQLite / 内存图 / simulated），绝不阻断启动或执行管道。
  • 幂等自初始化：需 FastAPI lifespan 初始化的组件（网关管理器）在 API handler 内以 ensure_ready() 幂等守卫首调自初始化，兼容「寿命周期未跑」的调用场景。
  • 事实锚点铁律：调参 / 优化只改策略阈值或实体关系，绝不改写业务数字或动作。
  • 收敛一致：V1-2 能力层联邦与 T4「收敛删死代码」方向一致，未新增 Agent 执行分支。

五、剩余范围（非本次发布）

  • 多 Agent 协作 / 编排：与 T4「收敛」方向相反，为既定非目标，V1 未纳入。
  • 多租户 / 企业版隔离（V2 范围）：暂无 tenant_id / plant_id 三级隔离。
